### Phase 0-3：`BEQZ` 指令格式、分支條件與完整手推

狀態：✅ 已完成

驗收證據：

- 能說明 `BEQZ` 三個欄位的用途：opcode、條件暫存器位址與 3-bit signed branch offset。
- 能正確判讀正、負 3-bit signed branch offset，並說明 sign extension。
- 能說明 `branch_en` 只在 `BEQZ` 的 EXECUTE 階段啟用零值比較路徑。
- 能說明 `branch_taken = branch_en && (reg1_data == 0)`。
- 能說明 `BEQZ` 的 `reg_write = 0`，不寫回 Register File。
- 能正確區分 branch taken 與 not taken。
- 已依目前 RTL 確認 branch target 使用目前 PC 作為基準：taken 時 `PC_next = PC + signed offset`；not taken 時 `PC_next = PC + 1`。
- 能完成 taken 與 not taken 的 FETCH／EXECUTE 手推。
- 能指出 FETCH 正緣更新 IR；EXECUTE 正緣更新 PC；`BEQZ` 不更新 Register File。
- 獨立驗收題：`BEQZ R3, -2 = 8'b10011110`；PC 為 50 時，taken 更新為 48，not taken 更新為 51。

限制：本紀錄只代表觀念與手推驗收通過；尚未以 self-checking testbench 驗證 branch taken／not taken，也未完成 Phase 0 整體通關。

## 下一個小階段

### Phase 0-4：`CP` 指令格式、資料複製路徑與完整手推

唯一目標：能獨立完成一條 `CP` 的編碼與反向解碼，說明來源／目的暫存器、Writeback MUX 的資料複製路徑，以及 FETCH／EXECUTE 的控制訊號與更新 edge。

預計驗收重點：

- `CP` 的 opcode 與三個欄位用途。
- `reg1_addr` 作為目的暫存器位址，`reg2_addr` 作為來源暫存器位址。
- `wb_data_sel = 1` 時 Writeback MUX 選擇 `reg2_data`。
- `reg_write = 1`，來源資料在 EXECUTE 正緣寫入 `reg1_addr`。
- `alu_src_sel` 與 ALU 結果對 `CP` 的實際影響。
- `branch_en = 0`、`branch_taken = 0`，PC 走 `PC + 1`。
- FETCH／EXECUTE 的主要控制訊號。
- IR、PC 與 Register File 的更新 edge。
- 一題獨立完成的 `CP` 編碼、資料流與時序驗收題。
